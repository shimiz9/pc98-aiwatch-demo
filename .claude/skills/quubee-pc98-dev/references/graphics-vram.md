# QuuBee上のグラフィックVRAM直接描画とMAG画像形式

PC-98標準デジタルグラフィックへの直接描画、および静止画像形式MAG(MAKI02)の
自作エンコードについて。ソース根拠はQuuBeeのサブモジュール
`core/np2kai`(AZO234/NP2kai)と`web/player/magimage.js`。

## グラフィックVRAMのプレーン配置

標準デジタル8色モード。`core/np2kai/i286c/cpumem.h`の`VRAM_B/R/G/E`定数で確認:
- セグメント`0xA800` = Bプレーン
- セグメント`0xB000` = Rプレーン
- セグメント`0xB800` = Gプレーン
- (`0xE000` = 輝度/Eプレーン。16色化には別途モード切替が必要そうで未検証)

各プレーンは1bpp、1行80バイト(640px/8)。色は3プレーンのビットON/OFFの組み合わせで
決まる固定デジタル8色(000=黒,001=青,010=緑,011=シアン,100=赤,101=マゼンタ,
110=黄,111=白)。Watcom Cでは`<i86.h>`の`MK_FP(seg,0)`でセグメント先頭への
far pointerを作り、`<string.h>`の`_fmemcpy`/`_fmemset`や直接のバイト代入で書き込む。

## 重要な注意点1: グラフィック画面は明示的に有効化しないと表示されない

**実機ならBIOSが起動時に自動でGDC(uPD7220)へSTARTコマンドを送るが、QuuBeeの
HLE-DOSはテキスト画面の表示だけを自動有効化し、グラフィック画面のGDC初期化は
プログラム任せ。** これを行わずにGVRAMへ書き込んでも画面には何も表示されない
(テキストは問題なく表示されるので、原因特定に時間がかかりやすい)。

**対策:** グラフィック用GDCのコマンドポート`0xA2`へSTARTコマンド(`0x6B`)を
一度送る:
```c
#include <conio.h>
outp(0xA2, 0x6B);
```
これで`gdcs.grphdisp`に`GDCSCRN_ENABLE`が立ち、以降の書き込みが表示されるようになる。
(根拠: `core/np2kai/io/gdc.c`の`gdc_work()`/`gdc_oa2()`、`vram/scrndraw.c`の
`GDCSCRN_ENABLE`チェック)

## 重要な注意点2: グラフィック画面は縦200ラインを2倍に引き伸ばして表示される

テキスト画面(25行×16px=400ライン、等倍)と異なり、**グラフィック画面は実効的に
縦200ライン相当のデータを2倍に引き伸ばして400ライン相当として表示する**挙動が
ある(PC-98の元々のグラフィックモードが200ライン機だったことに由来すると推測)。

そのため、グラフィックとテキストを同じ画面に重ねて使う場合、**縦方向の座標だけ
半分の値で計算する**必要がある(横方向は影響を受けない)。例えば「画面のy=113の
位置に描画したい」場合、実際にAPIへ渡す値は`y=56`(半分)にする。この座標系の
食い違いに気づかず座標を組むと、意図した位置より縦に大きく・下にズレて表示され、
他の描画要素と重なるバグになる。

**教訓:** グラフィックとテキストの座標系(特に縦方向のスケール)が同じとは
限らない。古い機種のグラフィックハードウェアは「本来低解像度だったものを
引き伸ばして高解像度機と互換に見せる」実装が多く、机上の計算より実機
(エミュレータ)で目視して補正する方が確実。

## MAG(MAKI02)画像形式の自作エンコード

構造(`web/player/magimage.js`のデコーダをリファレンスに確認):
```
"MAKI02  "(8B) + コメント(0x1A終端) + 32Bヘッダ + パレット + flagA + flagB + pixel
```
- ヘッダ(0x1Aの次から32B): +3=screenMode(bit7=256色/bit0=200ライン)
  +4/6/8/10=x0/y0/x1/y1(word) +12/16/24=offA/offB/offP(dword、ヘッダ起点の相対オフセット)
- パレットはG,R,B順。16色=48B/256色=768B
- units(1ユニット=4byte出力)/lines等でサイズが決まる。16色ならshift=3
  (`units=(x1>>3)-(x0>>3)+1`)

**エンコードを単純化するコツ: flagAを全ビット0にすれば無圧縮になる。**
MAGは行/列をまたいだ差分コピー(flagA/flagBによるRLE的圧縮)に対応しているが、
flagAが全て0なら「毎回ピクセルストリームから生データを読む」動作になり、
ピクセルストリームは単純な4bpp(1バイト=2ピクセル、上位ニブル=左)パックの
raw画像データそのものになる。圧縮ロジックの実装が不要になり、ファイルサイズは
最適でなくても100%正しくデコードされる。

**検証方法:** デコード側(magimage.js)のロジックをPythonで再実装した検証
スクリプトを書き、自作エンコーダの出力をデコード→画像化して元画像と目視
比較すると、QuuBee実機での確認サイクルを1回節約できる。
