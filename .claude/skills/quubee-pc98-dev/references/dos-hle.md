# QuuBee HLE-DOS: テキスト表示・ビルド・文字コードの落とし穴

QuuBeeはNEC BIOS/MS-DOSを実装せず、`native/dos_int21.c`でINT 21hの必要な
サブセットだけをC側で再実装している(HLE)。ソース: https://github.com/msonrm/quubee

## サポートされるINT 21hサブセット

`native/dos_int21.c`で確認済み。実用上必要な範囲は一通り実装されている:
- テキスト表示: AH=00h-0Eh(直接コンソール出力、カーソル制御、ANSI/PC-98 CONのESCシーケンスも解釈)
- 文字/文字列入出力: AH=01h,06h,07h,08h,09h,0Ah,0Bh(入力バッファ確認),0Ch
- ファイル操作: AH=3Ch/3Dh/3Eh/3Fh/40h/41h/42h/43h/45h/46h/4Eh/4Fh
- メモリ: AH=48h/49h/4Ah/58h
- プロセス: AH=4Bh(exec)/4Ch(exit)/4Dh/31h(TSR)
- 時刻: AH=2Ch(get time) — 正確な実時間が必要な処理(後述)に使える
- その他: 19h/1Ah/25h/35h/2Ah/30h/36h/39h-3Bh

EXEロードは`magic==0x5A4D(MZ)||0x4D5A(ZM)`で判定(`native/dos_loader.c`)。
**MZ形式.EXE、16bitリアルモード、DOSエクステンダ不使用**が前提。COM形式も対応。

不明な仕様は`dos_int21.c`を`grep`して確認するのが最も確実(公式ドキュメントより正確)。

## ビルド時の文字コード: UTF-8ソース → CP932変換が必須

ソースをUTF-8で編集・保存しても、`wcl`はバイト列をそのまま文字列リテラルとして
埋め込むだけなので、実行時にQuuBee(CP932/Shift-JIS前提のDOS環境)がUTF-8バイト列を
CP932として誤解釈し、日本語が文字化けする。

**対策:** コンパイル直前にソースをCP932へ変換してからビルドする。
`scripts/build.sh`がこれを自動化している(`iconv -f UTF-8 -t CP932`)。

**注意点:**
- `-t SHIFT-JIS`より`-t CP932`の方がNEC/IBM拡張文字を含め網羅性が高く安全。
- em dash(—)や≈等のUnicode専用記号はCP932に存在せず変換エラーになる。
  日本語混じりのソースコメントであっても、半角ハイフン`-`・`->`・「約」等の
  ASCII/CP932確実対応の表現に置き換えること。
- ビルドスクリプトが`set -e`でも、パイプ(`| tail`等)経由で呼ぶと呼び出し元の
  `&&`判定はパイプ全体の終了コードを見てしまい上流の失敗を見逃す。生成物の
  タイムスタンプ/サイズでビルド成否を確認する習慣があるとよい。

## AH=09h(文字列表示)の`$`終端忘れバグ

DOS AH=09hはDS:DXから`'$'`(0x24)バイトが見つかるまでメモリを読み進めて表示する。
`dprint("\r\n\r\n")`のように**`$`を付け忘れる**と、リンカが直後に配置した
**次の文字列リテラルまで読み進めて巻き込んで表示してしまう**(次の行が二重に
見える等の不可解なバグになる)。

**検出方法:** `grep -n 'dprint(' src/*.c | grep -v '\$");'` で機械的に検出できる。
空行だけを出力する意図の呼び出し(`dprint("\r\n\r\n")`)ほど「文字列っぽくない」ので
$を付け忘れやすい。

## カーソル位置の保存/復元(ESC[s / ESC[u)

QuuBeeのtty_putc(AH=09h/02h等が共有する出力ルーチン)はANSI/PC-98 CONのESC
シーケンスを解釈する。対応コマンド(`native/dos_int21.c`の`csi_dispatch`):
- `ESC[2J` 画面全消去+カーソルホーム、`ESC[nJ`/`ESC[nK` 部分消去
- `ESC[<row>;<col>H`/`f` カーソル位置(1-based)
- `ESC[nA/B/C/D` カーソル移動
- `ESC[s`/`ESC[u` **カーソル位置(+属性)の保存/復元**

「一時的に別の場所へ描画してから元の処理へ戻る」サブルーチン(帯状スクロール等)は
**必ず`ESC[s`/`ESC[u`でカーソル位置を保存・復元すること**。しないと、描画後の
カーソル位置に呼び出し元の後続出力が続いてしまい、画面最下行での改行時に
画面全体がスクロールして過去の描画が残像のように積み上がるバグになる。単体テストでは
何も後続出力がないため発覚せず、他の画面要素と組み合わせる統合時に初めて顕在化する
典型パターン。

## 全角/半角混在テキストのセンタリング

全角文字(かな・カナ・漢字)は半角の2倍の表示幅を持つため、目分量でスペース数を
決めると必ずズレる。Pythonの`unicodedata.east_asian_width()`で全角/半角を判定し、
画面幅(通常80桁)を基準に必要なインデント量を計算すること:

```python
import unicodedata
def width(s):
    return sum(2 if unicodedata.east_asian_width(c) in ('W', 'F') else 1 for c in s)
def center_indent(s, screen_width=80):
    return max(0, (screen_width - width(s)) // 2)
```

## 帯状スクロール(横スクロールテロップ)は半角カナで構成する

全角(2バイトSJIS)文字を1バイトずつずらして描画すると、ウィンドウ境界が
2バイト文字の途中に来た瞬間に上位/下位バイトが分離して文字化けする。
**半角カナ+半角英数のみ(CP932で1バイト=1桁)で構成すれば、バイト単位の
スクロールでも崩れない。** これは実機PC-98時代のデモでも定番の手法。

半角カナ変換の確認:
```bash
echo 'ｱｲｳｴｵ ﾄﾞｳｺｳ' | iconv -f UTF-8 -t CP932 | xxd   # 各文字が1バイトになるか確認
```

実装は、メッセージをリング状に見立てて画面幅ぶんの窓を1バイトずつスライドさせ、
`ESC[<row>;<col>H`でカーソルを固定行へ移動してから毎フレーム上書きする方式でよい。
