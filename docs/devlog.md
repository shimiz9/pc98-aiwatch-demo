# 開発ログ — AI Watch Intro (PC-98 FM音源デモ)

生成AI(Claude)との対話を通じてQuuBee向けPC-98デモを作る過程の記録。
プロンプト、判断、うまくいかなかった点を時系列で残す。

## 2026-07-02 フェーズ1: 開発環境の統合

以前のセッション(workspace直下)で確立済みだったOpenWatcom V2環境をそのまま流用。
- `openwatcom/` (16bit DOSクロスコンパイラ一式、`open-watcom-v2`のsnapshotビルド)はworkspaceルート(`/home/shimiz/workspace/openwatcom`)に置いたまま共有し、プロジェクト側の`tools/build.sh`から絶対パスで`owsetenv.sh`をsourceする方式にした。
  - 判断理由: OpenWatcom一式は約270ファイル・相当なサイズがあり、プロジェクトごとにコピーすると各リポジトリが肥大化する。複数プロジェクトで共有する前提なら絶対パス参照で十分。
- `tools/build.sh src/xxx.c OUT.EXE` で `wcl -bt=dos -ms` によるコンパイル+リンク、distへのzip化まで一括実行。
- 動作確認: `src/hello_check.c`(INT 21h AH=09hのみ)をビルドし、`dist/CHECK.EXE`が `4D 5A`(MZ)ヘッダを持つ16bit DOSリアルモードEXEであることを`file`/`xxd`で確認。問題なし。確認後ファイルは削除(スケルトンのみ本採用)。

## 2026-07-02 フェーズ2: QuuBeeのHLE-DOS実装調査

`github.com/msonrm/quubee` (native/配下)と、そのサブモジュール `core/np2kai` (AZO234/NP2kai)をシャローcloneして調査。

### 使用可能なINT 21hサブセット (`native/dos_int21.c`)
文字表示・キー入力・ファイルI/O・メモリ確保・EXEC・終了系まで、実用上必要な範囲は一通り実装されている:
- テキスト表示: AH=00h〜0Eh (直接コンソール出力、カーソル制御、ESC/CSI相当のANSI風エスケープも解釈)
- 文字/文字列I/O: AH=01h,06h,07h,08h,09h,0Ah,0Bh,0Ch (getch/putstr等)
- ファイル操作: AH=3Ch(create)/3Dh(open)/3Eh(close)/3Fh(read)/40h(write)/41h(delete)/42h(seek)/43h(attr)/45h,46h(dup)/4Eh,4Fh(findfirst/next)
- メモリ: AH=48h(alloc)/49h(free)/4Ah(resize)/58h(alloc strategy)
- プロセス: AH=4Bh(exec)/4Ch(exit)/4Dh(retcode)/31h(TSR keep)
- その他: AH=19h(カレントドライブ)/1Ah(DTA設定)/25h,35h(割り込みベクタ)/2Ah,2Ch(日付時刻)/30h(DOSバージョン)/36h(空き容量)/39h〜3Bh(ディレクトリ操作)
→ **本デモが必要とする範囲(ファイル読み込み・テキスト表示・終了)はすべてカバーされている。**

### EXEロード対応 (`native/dos_loader.c`)
`magic == 0x5A4D (MZ) || 0x4D5A (ZM)` でEXE形式と判定するローダーが存在(1762行目, 1974行目付近)。`.COM`形式(拡張子で判定)にも対応。
→ **`.EXE`(MZ形式)での配布方針は問題なし。**

### OPNA (FM音源) のI/Oポート
QuuBeeの`native/qb_soundmng.c`はJS側WebAudioへのPCM受け渡しグルー(pull型)のみで、OPNAレジスタのエミュレーション自体はサブモジュール`core/np2kai`(NP2kai本体、fmgenベース)が担当。HLE対象ではなく、**実機同様のハードウェアI/Oエミュレーション**である点を確認できたのは重要な収穫:
- `core/np2kai/cbus/board86.c`: オンボード音源として`0x188`固定ベースでOPNAを登録(`board86_bind`内、`g_opna[0].s.base = 0x000; // 0x188固定`)
- ポート内訳(`opna_o188`/`opna_o18a`/`opna_o18c`/`opna_o18e`などの関数名から確認):
  - `0x188` = レジスタセットA アドレス書き込み / ステータス読み出し
  - `0x18A` = レジスタセットA データ書き込み / データ読み出し
  - `0x18C` = レジスタセットB(拡張チャンネル)アドレス書き込み
  - `0x18E` = レジスタセットB データ書き込み
→ **`outp(0x188, addr); outp(0x18A, data);`のような直接ポートI/Oでよく、市販ドライバは一切不要。この点は当初の想定通りで、実装方針が裏付けられた。**

### テキストVRAM/グラフィックVRAM
`native/qb_scrnmng.c`はJS Canvas側へのサーフェス受け渡しのみで、PC-98のテキストVRAM(0xA000セグメント)・グラフィックVRAM(0xA800/0xE000)自体のメモリマッピングはHLE対象外、`core/np2kai`のメモリコアが実機同様にエミュレートしている前提。
→ 次フェーズでfar pointer(`MK_FP`)による0xA000:0000への直接書き込みが実機同様に機能するか、実際にビルドしたEXEで確認する必要あり(未検証)。

### 試行錯誤・つまずいた点
- `core/np2kai`はgit submoduleのため、最初の`git clone --depth 1`だけでは空ディレクトリになった。`.gitmodules`を確認し`git submodule update --init --depth 1 core/np2kai`で追加取得が必要だった。
- OPNAのI/Oポート実装自体はQuuBee本体(native/)ではなくNP2kai本体(core/np2kai/cbus/board86.c)にあり、最初`native/`配下だけを探して見つからなかった。QuuBeeが「DOSシステムコールだけHLE、ハードウェアエミュレーションはNP2kai本体そのまま」というアーキテクチャだと理解してから探索先を変えたら見つかった。

## 2026-07-02 フェーズ3: 文字化けトラブルとその修正

ユーザーに手動でQuuBeeへドラッグ&ドロップして動作確認してもらったところ、タイトルロゴ(ASCIIアート)は正常に表示されたが、日本語部分(サブタイトル・"Press any key"の日本語注記など)が文字化けした。

**原因:** ソース`src/aiwatch.c`はUTF-8で編集・保存していたが、`wcl`はソースのバイト列をそのまま文字列リテラルとして埋め込むだけで、実行時にQuuBee(PC-98 DOS環境)が期待するのはShift-JIS(CP932)。UTF-8のバイト列をそのままCP932として解釈したため文字化けした。

**修正:** `tools/build.sh`で、コンパイル直前に`iconv -f UTF-8 -t CP932`でソースを変換した一時ファイル(`build/`以下)を生成し、そちらをビルド対象にするよう変更。

**さらにつまずいた点:** 最初`-t SHIFT-JIS`を指定したところ、コメント中のemダッシュ「—」(U+2014)が変換できず`iconv: illegal input sequence`でエラー終了 → `set -e`でスクリプトは異常終了するはずが、パイプ(`| tail`)経由で実行していたため呼び出し元のシェルはエラーに気づかず、**古い(文字化けする)dist/AIWATCH.EXEをそのまま再利用してしまっていた**。異常終了に気づけたのは`build/aiwatch.c`のファイルサイズが不自然に小さい(18バイト)ことに気づいたため。
- 教訓1: 日本語混じりのソースをレトロ環境向けにビルドする際は、em dash・矢印(→)等のUnicode専用記号を混ぜない(半角ハイフン`-`や`->`で代替する)。
- 教訓2: `-t SHIFT-JIS`より`-t CP932`の方がNEC/IBM拡張文字を含め網羅性が高く、変換エラーが起きにくい。
- 教訓3: `set -e`を仕込んだスクリプトでも、パイプの下流コマンド(`tail`等)経由で呼ぶと呼び出し元の`&&`判定はパイプ全体の終了コード(=下流コマンドの終了コード)を見てしまい、上流の失敗を見逃す。ビルドスクリプトの成否は生成物のタイムスタンプ/サイズなど別の手段でも確認すべき。

## 2026-07-02 フェーズ3: タイトルロゴの崩れと「行の二重表示」バグ

文字化け修正版をユーザーに再度手動確認してもらったところ、日本語は正しく表示されたが、
1) ASCIIアートの"AI WATCH"ロゴがどの文字か判読できないほど崩れていた
2) サブタイトル行「生成AI・最新テクノロジー総合情報サイト」が、字下げ位置の異なる2行として重複表示される

という2つの問題が見つかった。

**1) ロゴ崩れの原因:** 手打ちで`#`を並べたASCIIアートの列位置がずれており、文字として成立していなかった。
**対応:** Pythonで5x5/5x7ブロックフォント(A/I/W/T/C/H)を辞書定義し、文字列連結で正確に整列させたものをC言語の`dprint`呼び出しへ機械的に変換して差し替えた。手打ちのドット絵は目視だけでは崩れに気づきにくく、生成AIによるプログラム的な整形の方が確実だった。

**2) 行の二重表示の原因(重要):** `dprint("\r\n\r\n");`のように、**`$`終端記号を付け忘れた`bdos(0x09,...)`呼び出しが3箇所**あった。DOS AH=09hはDS:DXから`$`(0x24)バイトが見つかるまでメモリを読み進めて表示し続ける仕様のため、`$`を忘れた文字列は、リンカがメモリ上で直後に配置した**次の文字列リテラル(=次のサブタイトル行)まで読み進めてそのまま表示してしまい**、その直後に本来のdprint呼び出しでも同じ文字列を表示するため、結果として二重表示になっていた。
- 教訓: AH=09h(`$`終端文字列表示)を使うヘルパー関数を自作する場合、呼び出し側任せにすると"$"の付け忘れに気づきにくい。`grep -n 'dprint(' src/*.c | grep -v '\$");'`のような機械的チェックを都度走らせて検出するのが確実。空行だけを出力する意図の呼び出し(`dprint("\r\n\r\n")`)ほど「文字列っぽくない」ので$を付け忘れやすい、という点も分かった。

## 2026-07-02 フェーズ3: センタリングのずれ修正

行の二重表示が直った状態で再度ユーザーに確認してもらったところ、ロゴは正しく表示されたが、サブタイトルや"Presented by..."行のセンタリングが画面中央からずれていた。

**原因:** 全角文字(かな・カナ・漢字)は半角文字の2倍の表示幅を持つが、字下げのスペース数を目分量で決めていたため、実際の表示幅(全角=2桁/半角=1桁)を考慮した中央揃え計算になっていなかった。

**対応:** Pythonの`unicodedata.east_asian_width()`で各文字を全角/半角判定し、画面幅80桁を基準に必要なインデント量を計算するスクリプトを書いて、タイトル画面・エンディング画面の各行を機械的に再計算・差し替えた。
- 教訓: 日本語混じりのレトロ環境向けテキストレイアウトは、目分量でのスペース調整だと必ずズレる。全角=2/半角=1の表示幅ルールをコードで計算し、インデントも生成AIに計算させる方が確実。

## 2026-07-02 フェーズ4: OPNA直接I/O発音の最小サンプル

`src/opna.h`/`src/opna.c`として、市販/フリーDOSドライバに依存しない自作の最小OPNAレジスタ制御モジュールを作成。
- ポート: `0x188`(regA addr)/`0x18A`(regA data)/`0x18C`(regB addr)/`0x18E`(regB data)。フェーズ2の調査(`core/np2kai/cbus/board86.c`)通りで動作。
- `channel`引数(0-5)からレジスタセットA/B・セット内チャンネル番号・キーオンレジスタ(0x28)向けチャンネルコード(4-6は`0x04|ofs`)を算出するヘルパーで、6チャンネル分をまとめて扱えるようにした(BGM実装時に流用予定)。
- 音色は最小構成としてアルゴリズム7(4オペレータ全キャリア加算)を使い、スロット1-3をTL=127で無音化、スロット4だけを鳴らすシンプルな単一正弦波相当の音にした。エンベロープ(AR/DR/SR/SL/RR)は経験値で仮設定。
- Fnum/Blockは `Fnum = freq * 144 * 2^(20-Block) / MasterClock` (MasterClock=7987200Hz)の式で算出。A4(440Hz)、Block=4で Fnum=520。

`src/opnatest.c`でchannel0からA4を鳴らすだけの検証プログラムをビルド。

### ビルドスクリプトを複数ファイル対応に変更
opna.cとopnatest.cの2ファイルを1つのEXEにリンクする必要が出たため、`tools/build.sh`の引数仕様を`<ソース.c> [出力名.EXE]`から`<出力名.EXE> <ソース.c...>`(複数可)へ変更。あわせて、CP932変換を「指定されたファイルだけ」から「src/配下の.c/.hを全部build/へミラー変換」する方式に変えた。理由: `opna.c`が`#include "opna.h"`のように相対パスでヘッダを参照するため、変換後の`build/opna.c`が同じ`build/`ディレクトリ内に変換済み`opna.h`を見つけられる必要があったため。

### つまずいた点
`src/opnatest.c`のコメントに使った近似記号「≈」(U+2248)がCP932に存在せず、`iconv: illegal input sequence`で変換失敗。フェーズ3のemダッシュ「—」の件と同様のパターン。
- 教訓(蓄積): 日本語コメントでも、数式記号(≈,→,±,×,÷等)はASCII代替("約","->","+-"等)にするか、少なくとも都度`iconv -f UTF-8 -t CP932 file > /dev/null`で全ソースを事前チェックする習慣が必要。今回のように未対応記号1つでビルド全体が(サイレントに古い成果物を使い回す形で)止まりかねないため、ビルドスクリプト側でも変換失敗を`set -e`で即座に検知できる構成にしておいてよかった。

音自体がQuuBee上で実際に鳴るかは、ユーザーによる手動確認待ち。

## 2026-07-02 フェーズ5: 帯状スクロール表示の実装

`src/scroll.h`/`src/scroll.c`で画面最下行(25行目)に流れるテロップを実装。`src/scrolltest.c`で単体検証。

**設計判断: スクロール文字列は半角カナ+半角英数のみで構成する。**
全角(2バイトSJIS)の漢字・カナを1バイトずつ横にずらして描画すると、ウィンドウの境界がちょうど2バイト文字の途中に来た瞬間に上位バイトと下位バイトが分離され、その1コマだけ文字化けする。半角カナ(CP932で1バイト=1桁の0xA1-0xDF)なら「1文字=1バイト=1桁」なので、バイト単位でスクロールしても崩れない。
→ これは実機PC-98時代のデモ・ソフトでも半角カナの帯状スクロールが定番だった実務上の理由と一致しており、期せずして「当時のパソコンショップ店頭デモ風」という要件にも合致した。

キャッチコピーは「AIの最新動向を届ける総合情報サイト」を半角カナに変換して`"AI ﾉ ｻｲｼﾝ ﾄﾞｳｺｳ ｦ ﾄﾄﾞｹﾙ ｿｳｺﾞｳ ｼﾞｮｳﾎｳ ｻｲﾄ｡"`とした。半角カナ変換はPython(`unicodedata`は正規化の向きが逆のため使わず、全角カナ→半角カナの対応表を手で当てはめて確認)。

実装は、メッセージをリング状に見立てて70桁ぶんの窓を1バイトずつスライドさせ、`ESC[25;5H`でカーソルを最下行へ固定してから毎フレーム上書きする方式。カーソル位置指定(`ESC[<row>;<col>H`)はフェーズ3で使った`ESC[2J`と同じくQuuBeeのANSI/CSIパーサ(`native/dos_int21.c`の`csi_dispatch`)で正式にサポートされていることをソースで確認済み。

## 2026-07-02 フェーズ6: オリジナルBGM(3パート)の実装

ユーザーと相談し、方向性を「軽快ポップ調(明るい長調・120-140BPM程度)」「3パート(メロディ+ベース+コード)」に決定。

**音名→Fnum/Blockテーブルの自動生成:** `tools/gen_notes.py`で、C2〜C6の49音についてフェーズ4で導出した式(`Fnum = freq*144*2^(20-Block)/MasterClock`)から`src/notes.h`/`src/notes.c`を機械生成する方式にした。音楽データ側は`NOTE_C4`のような音名マクロで書けるので、素の数値を並べるより検証・作曲がしやすい。

**曲構成:** C-G-Am-F(いわゆる王道進行)の4小節ループ。
- メロディ(ch0): 8分音符主体で跳ねる旋律。各小節のコードトーン+経過音で構成。
- ベース(ch1): ルート-ルート-5度-ルートの四分音符パターン(いわゆるブンブン系の弾むベース)。
- コード(ch2): 各小節のコードの3度を全音符で伸ばす。単一チャンネルにつき同時発音数は1音までという制約(下記)があるため、ベースの根音+5度と、コード担当の3度を組み合わせて三和音相当をチャンネル間で分担する構成にした。

**音色設計で気づいた制約:** OPNAのFMオペレータで「キャリア」として使うスロットのMUL(周波数逓倍)を変えると、そのスロット自体の発音ピッチがずれてしまう。音色ごとに違う音を出したくても、フェーズ4のようにスロット4だけをキャリアとして使う最小構成では「MULで音色を変える」という発想は使えない。そのため音色の違いは全てAR/DR/SR/SL/RR(エンベロープ)とTL(音量)だけで作ることにした(`opna_set_patch()`として一般化)。メロディ=速いアタック+短めのディケイでプラック感、ベース=最大音量+パンチのある減衰、コード=やや遅いアタック+長いリリースで控えめなパッド、という3種の音色に。

**プレイヤー実装:** `src/music.c`に、8分音符単位の「ティック」を基準に3トラックを同時進行させる簡易シーケンサを実装。各トラックは`{note, duration}`イベント配列を巡回参照し、残りティックが0になったら前の音をノートオフ→次の音をノートオン、という単純な仕組み(ドラムマシン的なイベントキュー方式)。

**テンポ調整:** QuuBee実機で聴いてもらったところ「ゆったりした感じ」で遅すぎるとのフィードバック。`TICK_WAIT`(暫定600000ループ)を2.4倍速の250000へ変更したところ「ちょうどいい」で確定。
- 対応の過程で`TICK_WAIT`を`musictest.c`のローカル定数から`music.h`の`MUSIC_TICK_WAIT`マクロへ格上げした。理由: フェーズ8で`aiwatch.c`本体に統合する際、テスト用exeとタイトルデモ本体とでテンポがズレないよう、チューニング済みの値を1箇所で管理したかったため。
- 教訓: ソフトウェアウェイトによるテンポは実機(エミュレータ)でのCPU速度に依存し、机上の計算(132BPM相当を狙って算出)は全くあてにならなかった。結果的に机上計算値の半分以下(600000→250000)が正解だった。次に似た調整をする時は、最初から「聴いて合わせる」前提で、計算値ではなく控えめな値から始めて速める方向で調整する方が手数が少なそう。

## 2026-07-02 フェーズ7: ロゴMAG画像と変換スクリプト

**仕様調査:** MAG(MAKI02)形式の仕様書を新たに探す代わりに、QuuBee自体のデコーダ実装`web/player/magimage.js`をリファレンスとして読んだ。QuuBeeで表示できることが最終目標なので、「QuuBeeが読めるバイト列」を直接の正解とするのが最も確実だと判断した。

**エンコード方針:** MAGは行/列をまたいだ差分コピー(flagA/flagBによるRLE的圧縮)に対応しているが、実装を単純化するため`tools/png2mag.py`では**flagAを全て0にする**(=毎回ピクセルストリームから生データを読ませる、無圧縮)方式にした。この場合、ピクセルストリームは単純な4bpp(1バイト=2ピクセル)パックのraw画像データそのものになり、圧縮ロジックの実装が不要になる。ファイルサイズは最適ではないが(ロゴ画像で10.6KB)、デコード仕様上100%正しく表示される。

**依存関係について:** PNGの読み込みと16色量子化にはPillow(pip)を使用。「変換ツールは自作」の要件は市販/非公開のMAG変換ツールに依存しないことが主旨と判断し、MAGのバイナリエンコード自体(ヘッダ・パレット・flagA・pixelストリームの構築)は完全に自前実装、PNGデコードのみ公開ライブラリに任せる形にした。

**自己検証:** QuuBee上でいきなり確認する前に、`magimage.js`のデコードロジックをPythonで再実装した検証スクリプトを書いて、生成した`gfx/AIWATCH.MAG`をデコード→PNG化し、元画像と目視比較した。完全一致を確認できたため、ユーザーには「QuuBee実機でも問題なく表示されるはず」という高い確度で確認を依頼できる。
- 教訓: バイナリフォーマットを自作パーサ/エンコーダで扱う時、相手側(デコーダ)の実装がソースとして読める場合は、それをそのまま「仕様書」として使い、さらに同じロジックをこちら側でも実装して自己検証することで、ユーザーの実機確認サイクルを1回節約できる。

**成果物:** `tools/make_logo.py`(ロゴのPNGソースを生成) → `gfx/logo_src.png` → `tools/png2mag.py`で`gfx/AIWATCH.MAG`(320x64, 16色)に変換。

## 2026-07-02 フェーズ9: 統合後バグ「スクロール行が画面を押し上げる」

ユーザーに通し確認してもらったところ、トピック紹介画面で過去のスクロール行が画面上に何本も残像のように積み上がり、内容が上へ流れ去ってしまう不具合が見つかった。

**原因:** `scroll_step()`は`ESC[25;5H`で毎回カーソルを最下行(25行目)へ移動してからスクロールテキストを描画していたが、描画後にカーソル位置を元に戻していなかった。そのため、`scroll_step()`の直後に呼び出し元(`aiwatch.c`のトピック紹介ループ)が`dprint("...\r\n\r\n$")`のようにテキストを続けて出力すると、そのテキストは「25行目の続き」から出力される。25行目(画面最下行)で改行すると、コンソールは通常の端末と同様に画面全体を1行分スクロールして次の行を作る。これが繰り返されることで、過去のスクロール行が消えずに画面上へどんどん押し上げられていくように見えた。

**対応:** `scroll_step()`の描画を`ESC[s`(カーソル位置保存)と`ESC[u`(復元)で挟み、呼び出し元のカーソル位置を一切変更しないようにした。`ESC[s`/`ESC[u`はフェーズ2の調査時点で対応を確認していた機能で、伏線を回収する形で解決できた。
- 教訓: 「一時的に別の場所へ描画してから元の処理へ戻る」ようなサブルーチンは、カーソル位置という暗黙のグローバル状態を必ず保存・復元すること。今回は単体テスト(`scrolltest.c`)ではスクロール行以外に何も出力しない構成だったため発覚せず、複数画面要素を組み合わせる統合フェーズで初めて顕在化した典型例だった。

## 2026-07-02 フェーズ10: グラフィックロゴの追加(完成後の追加要望)

完成後、ユーザーから「テキストのみだと寂しい、画像を描画できないか」との要望。テキストアートやMAG単体表示では飽き足らず、EXE本体がその場でグラフィック画面に絵を描く機能を新たに実装することになった。フェーズ7で「スコープが大きくなるので見送る」と判断していた領域に、結局後から踏み込む形になった。

### GVRAMプレーン配置の特定
NP2kaiソースを調査し、PC-98標準デジタルグラフィックのプレーン配置を特定:
- セグメント`0xA800`=Bプレーン, `0xB000`=Rプレーン, `0xB800`=Gプレーン(いずれも1bpp、1行80バイト)。`i286c/cpumem.h`の`VRAM_B/R/G/E`定数で確認。
- 色は3プレーンのビット組み合わせで決まる固定デジタル8色(黒/青/緑/シアン/赤/マゼンタ/黄/白)。輝度(4枚目のE)プレーンは追加の16色化コマンドが必要そうだったため今回は見送り、8色の範囲で構成した。

### つまずいた点1: 何も表示されない
上記プレーンへ直接書き込む最小テスト(`gvramtest.c`)を作っても、QuuBee上で完全に無反応(何も描画されない)だった。
- 原因調査でNP2kaiの`io/gdc.c`を読み込み、`gdcs.grphdisp`に`GDCSCRN_ENABLE`ビットが立っていないと画面合成(`vram/scrndraw.c`)でグラフィック層がスキップされる仕様と判明。
- さらに調査すると、`GDCSCRN_ENABLE`はGDC(uPD7220)への`START`コマンド(オペコード`0x6B`等)を送った時にだけ立つ。実機では起動時にBIOSが自動でこれを送るが、QuuBeeのHLE-DOSはテキスト画面の表示だけを直接有効化し、グラフィック画面のGDC START送出まではエミュレートしていないため、**プログラム側で明示的にSTARTコマンドを送る必要がある**と判明。
- 対応: `gvram_enable()`として`outp(0xA2, 0x6B)`(グラフィックGDCのコマンドポートへSTART送出)を追加。これで初めて画面に何か表示されるようになった。
- 教訓: 「実機なら自動でやってくれる初期化」を、HLE環境は必ずしも肩代わりしてくれるとは限らない。テキストは動くのにグラフィックだけ動かない、という現象自体が「テキストとグラフィックで初期化の主体が違う(一方はHLE側、一方はプログラム側の責任)」という設計上の手がかりだった。

### つまずいた点2: 色が縦縞に崩れる
表示はされるようになったが、複数色のカラーバーテストで黒・白は正常なのに、それ以外の中間色(赤・青・緑・シアン・マゼンタ・黄)だけ縦縞状のノイズになる現象が発生。3プレーンの内容が正しく合成されていない(プレーン間で1ドット単位のズレがある)ことを示唆する症状だった。
- `sdraw.mcr`(3プレーンをRGB合成して最終的な画面バッファへ書き出す処理)まで読み進めたが、正確な原因(合成側のアドレッシング方式の詳細)の特定には至らず、ここは深追いを止めてユーザーに方針を確認した。
- ユーザーの回答:「多少色が違ってもいいので、今の状況でやってみる」との判断をもらい、これ以上のハード内部調査は打ち切り、実際にロゴ(マゼンタ背景バッジ+白文字)を描画してみたところ、**大きい塗りつぶし矩形では縞模様は発生せず、問題なく表示された**(フェーズ4のOPNAでも経験したが、今回も「実際に試したら小さいテストケースだけの問題で、実用上は困らなかった」パターン)。

### つまずいた点3: グラフィックとテキストの座標系のズレ(縦2倍問題)
ロゴを統合したところ、狙った位置よりも画像が縦に大きく・下にズレて表示され、下に配置したテキスト(サブタイトル等)と重なってしまった。
- 検証の結果、QuuBeeのグラフィック画面は実際には**縦200ライン分のデータを2倍に引き伸ばして400ライン相当として表示している**らしいと判明(PC-98の元々のグラフィックモードが200ライン機だったことに由来する挙動と推測)。テキスト画面(25行×16px=400ライン、引き伸ばしなし)とグラフィック画面とで、実効的な座標系(特に縦方向のスケール)が食い違っていたのが原因。
- 対応: ロゴの座標を生成するPythonスクリプト側で、縦方向のセルサイズだけを半分(横6px/縦3px)にして計算し直し、表示側の2倍引き伸ばしと相殺させた。結果、意図した位置・サイズで表示されるようになった。
- 教訓: グラフィックとテキストを同じ画面に重ねて使う場合、それぞれの座標系(スケール)が同じとは限らない。特に古い機種のグラフィックハードウェアは「本来は低解像度だったものを引き伸ばして高解像度機と互換に見せる」という歴史的経緯を持つ実装が多く、机上の計算だけで正しい座標を出すのは難しかった。実機(エミュレータ)で目視して補正する方が結局早かった。
